# LLM Wiki 구축 계획

## 1. 컨셉 한 줄 요약

**RAG의 대안**으로, LLM이 직접 작성·유지하는 영속적 마크다운 위키.
매번 원문에서 정보를 재추출하지 않고, 한 번 정리한 지식을 누적·교차참조하여 재사용한다.

**핵심 통찰**: 위키 유지의 가장 큰 장벽인 "북키핑(bookkeeping) 비용"을 LLM이 거의 0으로 만든다.
사람은 출처 선정·질문 방향 설계만 하고, 정리·갱신·일관성 유지는 LLM이 담당한다.

## 2. 3계층 아키텍처

```
┌─────────────────────────────────────────────┐
│ 3) Schema (설정층) — 사람이 설계            │
│    CLAUDE.md, 스키마/규칙/워크플로 정의     │
└─────────────────────────────────────────────┘
               ↓ 지시
┌─────────────────────────────────────────────┐
│ 2) Wiki (지식층) — LLM이 작성·유지          │
│    요약·개체·개념 페이지 + 교차참조         │
│    index.md, log.md, *.md                   │
└─────────────────────────────────────────────┘
               ↑ 읽기 전용 참조
┌─────────────────────────────────────────────┐
│ 1) Raw Sources (원본층) — 불변              │
│    기사, 논문, PDF, 이미지, 노트 …          │
└─────────────────────────────────────────────┘
```

- **Raw Sources**: LLM은 읽기만 한다. 절대 수정하지 않는다.
- **Wiki**: LLM이 만들고 갱신한다. 마크다운 파일 + 상호 링크.
- **Schema**: 사람이 운영 규칙을 정의한다 (CLAUDE.md / AGENTS.md 등).

## 3. 폴더 구조 (제안)

```
llm-wiki/
├── PLAN.md                  ← 이 문서
├── CLAUDE.md                ← LLM 운영 규칙 (스키마층)
├── sources/                 ← 1) Raw Sources (불변)
│   ├── articles/
│   ├── papers/
│   ├── notes/
│   └── INDEX.md             ← 원문 인덱스 (출처/날짜/URL)
├── wiki/                    ← 2) Wiki (LLM 작성)
│   ├── index.md             ← 모든 위키 페이지 목록 (카테고리별)
│   ├── log.md               ← 감사 추적: 언제 무엇이 변경됐는지
│   ├── concepts/            ← 개념 페이지 (e.g. transformer.md)
│   ├── entities/            ← 개체 페이지 (인물·기업·제품)
│   ├── topics/              ← 주제별 종합 페이지
│   └── summaries/           ← 원문별 1:1 요약 페이지
└── scripts/                 ← (선택) ingest/lint 자동화 스크립트
```

## 4. 핵심 워크플로

### 4.1 Ingest (수집)

새 원본이 `sources/` 에 들어왔을 때:

1. LLM이 원문을 읽고 `wiki/summaries/<slug>.md` 로 요약 페이지 생성
2. 요약에서 추출한 개념/개체에 대해 `wiki/concepts/`, `wiki/entities/` 의
   기존 페이지를 **갱신** (10~15개 페이지가 함께 업데이트되는 게 정상)
3. 새 개념/개체면 새 페이지 생성 + 교차링크 추가
4. `wiki/index.md` 와 `wiki/log.md` 갱신
5. `sources/INDEX.md` 에 원문 메타데이터 추가

### 4.2 Query (질의)

사용자가 질문하면:

1. `wiki/index.md` → 관련 페이지 후보 식별
2. 해당 페이지들을 읽고 합성, 필요시 원문 인용
3. 답변이 재사용 가치가 있으면 `wiki/topics/` 에 새 페이지로 저장 (지식 축적)

### 4.3 Lint (정합성 점검)

주기적으로:

- 모순되는 진술 탐지 (같은 사실에 대한 상충 기록)
- 고아 페이지 (어디서도 링크되지 않는 페이지) 식별
- 누락된 교차참조 (페이지 A가 B를 언급하지만 링크 없음)
- 오래된 페이지 (원문이 갱신됐는데 위키가 안 따라간 경우)

## 5. 페이지 작성 규칙 (초안)

모든 위키 페이지 상단에 frontmatter:

```yaml
---
title: <페이지 제목>
type: concept | entity | topic | summary
sources: [<원문 ID>, ...]   # 근거가 된 원본
updated: 2026-05-03
---
```

본문 규칙:
- 모든 사실 진술 끝에 `[^source-id]` 형태의 출처 표시
- 다른 페이지 언급 시 반드시 `[[wiki-link]]` 또는 마크다운 링크
- 추측·해석은 명시적으로 "추론:" 으로 표시

## 6. 기술 스택 (제안)

- **저장**: 평문 마크다운 + 로컬 git 저장소 (버전 관리 = 자동 감사 추적)
- **에디터/뷰어**: Obsidian, VS Code, 또는 단순 마크다운 뷰어
- **자동화**: Claude Code 가 ingest/lint 의 기본 실행 환경
  - `CLAUDE.md` 에 워크플로 명시 → 사용자가 자연어로 "이거 인제스트해줘" 하면 LLM 이 규칙 따라 처리
- **이식성**: 마크다운이라 모델/도구 갈아타도 그대로 살아남음

## 7. MVP 단계

**Phase 0 — 스켈레톤** (이번 단계에서 만들 것)
- [x] `llm-wiki/PLAN.md` 작성 (이 문서)
- [ ] 빈 폴더 구조 생성
- [ ] `CLAUDE.md` 초안: 페이지 규칙·워크플로·금지사항
- [ ] `wiki/index.md`, `wiki/log.md` 빈 파일 + 형식 예시

**Phase 1 — 첫 인제스트**
- 원문 1개를 `sources/` 에 넣고 끝까지 처리
- 요약 페이지 + 개념 페이지 2~3개 생성
- index/log 갱신 흐름 검증

**Phase 2 — 누적 검증**
- 같은 주제 원문 3~5개 더 추가
- 기존 개념 페이지가 제대로 "갱신"되는지 확인
- 한 번 정리한 지식이 다음 인제스트에서 재사용되는지 관찰

**Phase 3 — Lint & Query 정착**
- 주간 lint 루틴
- 자주 나오는 질문은 `wiki/topics/` 페이지로 승격
